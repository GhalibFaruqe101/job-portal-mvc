2. The Data Contract: Sessions (MANDATORY)

Each module will implement its own `login.php` / `AuthController.php`. On successful login you MUST set the following exact session keys (case-sensitive):

- `$_SESSION['user_id']` : The integer ID from the `users` table.
- `$_SESSION['role']` : The exact role string, one of `'seeker'`, `'employer'`, `'recruiter'`, or `'admin'`.

Do NOT use alternate names such as `$_SESSION['userid']` or `$_SESSION['logged_in']` — doing so will break inter-module communication.

3. The Templates (Copy these into your folders)
----------------------------------------------

A. Database Connection Template
--------------------------------
Create a file at `<your_role>/config/db.php` and use this exact code. If your local XAMPP has a password, change it locally but do NOT push credentials to GitHub.

```php
<?php
// Template: db.php
$db_host = "localhost";
$db_user = "root";       
$db_pass = "";           // CHANGE LOCALLY IF NEEDED, BUT KEEP EMPTY ON GITHUB
$db_name = "job_portal_db"; 

$conn = new mysqli($db_host, $db_user, $db_pass, $db_name);

if ($conn->connect_error) {
    die("Database Connection Failed: " . $conn->connect_error);
}
$conn->set_charset("utf8mb4");
?>
```

B. Session & RBAC Template
--------------------------
Create a file at `<your_role>/helpers/session.php`. Include this at the top of all protected views (dashboards, admin pages) to secure them.

```php
<?php
// Template: session.php
// 1. Start the session safely (prevents "Headers Already Sent" errors)
if (session_status() === PHP_SESSION_NONE) {
    session_start();
}
// 2. Standard Login Check
function is_logged_in() {
    return isset($_SESSION['user_id']) && isset($_SESSION['role']);
}
// 3. Role-Based Access Control (RBAC)
function require_role($required_role) {
    if (!is_logged_in()) {
        // Redirect to your specific login page
        header("Location: ../views/login.php"); 
        exit();
    }
    if ($_SESSION['role'] !== $required_role) {
        die("Access Denied: You do not have permission to view this page.");
    }
}
?>
```

Example usage for a dashboard (`seeker/views/dashboard.php`):

```php
<?php
require_once '../helpers/session.php';
require_role('seeker'); // Secures the page!
require_once '../config/db.php';
?>
<!DOCTYPE html>
<html>
<head>
    <!-- 1. Shared Global CSS -->
    <link rel="stylesheet" href="../../public/css/style.css">
    <!-- 2. Your Specific CSS -->
    <link rel="stylesheet" href="../../public/css/seeker/dashboard.css">
</head>
<body>
    <h1>Welcome User #<?php echo $_SESSION['user_id']; ?></h1>
</body>
</html>
```

# UI & Design System Guidelines

To ensure our platform looks like a single, cohesive application instead of four separate projects, we are using a **Shared Global CSS** file. 

## 1. The Global CSS (`/public/css/style.css`)
This file contains our brand colors, universal navigation bar styling, and standard input/button designs. It has already been pushed to the `main` branch. 

**Do not edit this file without notifying the team.** If you need specific styles for your module, create your own CSS file in your designated `/public/css/your_role/` folder.

## 2. The Master HTML Boilerplate
Because we cannot use a shared header/footer PHP file (per the project rubric), you **must** use the following HTML structure as the starting point for every view you build. 

This ensures the Global CSS links correctly and the navigation bar looks identical across all modules.

### Boilerplate Template:
Copy and paste this into your view files (e.g., `dashboard.php`).


## 1. The Global CSS (`/public/css/style.css`)

**Why is this needed?**
Since we are all building our HTML views independently, we risk creating a platform that looks like four completely different websites stitched together (the "Frankenstein" effect). The Global CSS acts as our single source of truth for the platform's "Brand Identity." It guarantees that no matter which module a user is in, the core design feels cohesive, unified, and professional.

**What components does it affect?**
To save time and maintain consistency, **do not write custom CSS for the following elements**—they are already handled globally by this file:
* **The Universal Navigation Bar:** The top menu layout, logo placement, and link hover states.
* **Typography & Base Layout:** The default font family, text colors, background colors, and CSS resets (margins/padding).
* **Primary Buttons:** Any button utilizing the `.btn-primary` class will automatically inherit the platform's standard branding, padding, and transition effects.
* **Form Inputs:** Text boxes, dropdowns, and labels utilizing the `.form-group` and `.form-control` classes will have unified padding, borders, and focus outlines.
* **Containers / UI Cards:** The `.card` class provides a standardized white container with rounded corners and a subtle shadow for wrapping your dashboard content.

**Let's follow this** Let's not edit this file without notifying the team in the group chat. If you need specific styles for your module (e.g., a custom table layout for applicants or a unique dashboard chart), create your own CSS file in your designated `/public/css/your_role/` folder and link it below the global stylesheet.

5. Git Workflow Rules
---------------------
- Never push directly to `main`.
- Work on your own branch, e.g., `feature/seeker-module`.
- Commit often with descriptive messages (e.g., "Added employer job posting form").
- Push your branch and open a Pull Request when ready for review.
- If you modify the shared `public/css/style.css`, announce it to the team so others can pull changes to avoid conflicts.
